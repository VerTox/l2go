# Дизайн: Intention-машина игрока в GameLoop

**Дата:** 2026-06-30
**Статус:** утверждён к планированию
**Связанные задачи:** l2go-2za (атака не добегает), l2go-8qy (диалог с NPC через раз), «движение нельзя отменить»

## Проблема

Auto-attack и взаимодействие с NPC срабатывают ненадёжно («через раз»): когда персонаж
добегает до цели, удар/диалог часто не начинаются, а начатое движение к цели нельзя
отменить кликом в сторону — только снятием таргета.

### Корневая причина (подтверждено логами и кодом)

Изначальная архитектура: GameLoop — единый владелец игровой логики; handlers и (в будущем)
AI складывают события в очереди, а луп раз в тик их обрабатывает. Эта модель реализована
для combat / NPC / EXP / interact (очереди `commands chan Command` + `events EventQueue`,
обработка в `tick()` раз в 100 мс).

**Движение игрока ушло из этой модели.** `handlers/client/movement.go` при
`MoveBackwardToLocation` / `ValidatePosition` / `StopMovement` напрямую вызывает
`movementUseCase.UpdatePosition/StartMovement/StopMovement`, синхронно записывая позицию
в registry **из горутины хендлера, минуя очередь GameLoop**. Команда `CmdPlayerMoved`
долетает до GameLoop, но используется лишь для активации регионов.

Следствие: combat-логика GameLoop (approach-поллинг `NextAttackEvent` /
`InteractApproachEvent`) читает позицию, которую асинхронно пишет другой компонент из
другой горутины. Позиция в registry обновляется только редкими `ValidatePosition`
(в логах — раз в ~5 секунд) и застывает между ними. Поэтому approach-поллинг каждые 300 мс
видит устаревшую позицию: 16 ретраев «вне радиуса», а когда «прибытие» наконец
срабатывает — клиент уже снова в движении, и действие не применяется.

Дополнительно отсутствует модель **intention** (намерения): нечему отменить движение к
цели при новой команде игрока.

## Решение

Достроить изначальную идею до игрока: вернуть движение игрока под GameLoop и ввести
intention-машину в стиле L2J (`L2CharacterAI`: `AI_INTENTION_*`, `maybeMoveToPawn`,
`onEvtArrived`, `changeIntention`).

GameLoop (single goroutine — без гонок) становится единым авторитетом позиции движущегося
игрока: тик пересчитывает позицию по серверной интерполяции, эмитит «прибытие», ведёт
намерение и выполняет действие при достижении цели. `ValidatePosition` от клиента
превращается из источника истины в коррекцию/анти-чит.

### Компоненты (все в пакете `gameloop`)

- **`Intention`** — enum: `IDLE, ACTIVE, MOVE_TO, ATTACK, INTERACT, CAST, FOLLOW`.
  `CAST` и `FOLLOW` — каркас-заглушки (системы скиллов/follow ещё нет).
- **`PlayerAIState`** (per-charID, как `combatState`): текущее `Intention`, цель
  (`TargetObjectID` или точка), параметры намерения, состояние движения
  (`IsMoving`, `MoveStarted`, `MoveDestination`, скорость).
- **`PlayerMover`** — серверный расчёт движения в тике: интерполирует позицию по
  скорости/направлению/времени (переиспользуя расчётные функции `movement usecase`),
  пишет позицию в registry, шлёт `MoveToLocation`/`MoveToPawn` клиенту (throttled ~1 c,
  как L2J `_moveToPawnTimeout`), детектит прибытие в attack/interact reach.
- **Команды от handlers:** `CmdMoveToLocation`, `CmdValidatePosition`,
  расширение существующих `CmdAttackRequest` / `CmdInteractRequest`. Все ставят intention
  через `setIntention`.

### Дистанция прибытия

Переиспользуется уже реализованная и покрытая тестами формула `attackReach` (L2J:
`base + collisionRadius игрока + collisionRadius цели`):
- атака: `base = physicalAttackRange` (пока 40, TODO weapon/POWER_ATTACK_RANGE);
- интеракция: `base = 36` (L2J `thinkInteract`), порог диалога — `INTERACTION_DISTANCE` (150).

### Потоки данных

```
клик по земле        → MoveBackwardToLocation → CmdMoveToLocation
                       → setIntention(MOVE_TO, точка), старт движения
ValidatePosition     → CmdValidatePosition → сверка дрейфа (анти-чит), коррекция при расхождении
двойной клик по NPC  → Action → CmdAttackRequest / CmdInteractRequest
                       → setIntention(ATTACK/INTERACT, target)
тик (100 мс)         → для каждого движущегося: интерполяция позиции, broadcast,
                       проверка прибытия (reach)
прибытие             → onArrived → выполнить intention (attack swing / open dialog / cast-заглушка)
новая команда        → setIntention перезаписывает текущее → старое движение/действие отменяется
```

### Авторитет позиции и `ValidatePosition`

Тик GameLoop — авторитет позиции движущегося игрока. `ValidatePosition` от клиента
сверяется с серверной позицией: при малом расхождении принимается (толерантность к
лагам/геодате), при большом — серверная коррекция (`ValidatePositionServer`, механизм уже
есть). Это устраняет «застывшую позицию» как источник истины для combat-проверок.

## Миграция `movementUseCase`

Не удаляется. Разделяется на «расчёт» и «состояние»:
- **Расчёт** (траектория, скорости, `calculatePositionAtTime`, валидация дрейфа) —
  остаётся чистыми хелперами, переиспользуется из тика.
- **Состояние** движения (`IsMoving`, `MoveStarted`, `MoveDestination`, позиция) —
  переходит под единоличное владение GameLoop; пишет только тик. Хендлеры перестают
  писать позицию напрямую и шлют команды в очередь.

## Фазовая декомпозиция (эпик)

Вертикальные срезы: каждый компилируется, тестируется и не ломает остальное.

| Фаза | Содержание | Чинит |
|---|---|---|
| 1. Движение под GameLoop | Перенос состояния и тиковой интерполяции позиции игрока в GameLoop; `ValidatePosition` → коррекция. Без intention. | застывшую позицию (фундамент) |
| 2. Intention-ядро | `Intention` enum, `PlayerAIState`, `setIntention`, `onArrived`; перевод attack/interact на intention; отмена по новой команде. | l2go-2za, l2go-8qy, «нельзя отменить» |
| 3. Чистка и каркас | Убрать старый approach-поллинг (ретраи `NextAttackEvent` / `InteractApproachEvent`); заготовки `CAST`/`FOLLOW`. | техдолг, фундамент на будущее |

## Тестирование

- **Юнит (TDD):** интерполяция позиции по времени/скорости; проверка прибытия с
  collision-reach (`attackReach` уже покрыт); переходы intention (setIntention,
  отмена/перезапись, onArrived).
- **Интеграция:** проверка движения, атаки и диалога в живом клиенте после каждой фазы
  (требует участия пользователя — клиент L2 запускается им).

## Решения по сегодняшним combat-фиксам

- `attackReach` / collision-логику **оставляем** — верна по L2J, нужна для проверки
  прибытия в фазе 2.
- Throttle-ре-отправку MoveToPawn убираем в фазе 1 — движением станет править тик.

## Не входит в объём (YAGNI)

- Реальная реализация скиллов (`CAST` — только enum-заготовка и точка ветвления).
- Реальный `FOLLOW` за движущейся целью (заготовка).
- Геодата/pathfinding (движение по прямой, как сейчас).
- NPC AI (отдельный эпик; intention-каркас на него рассчитан, но не реализуется здесь).
